The QA Testing Channel, es un grupo de consultores especializados en Prueba de Software 
localizados en diferentes países como USA, México, España e India, entre otros.

Aprendizajes esperados:
Las tareas que realiza un Ejecutor de Prueba de Software.
Las bases de prueba que debe tener un ejecutor de prueba.
Los puntos importantes requeridos en la industria para cubrir el perfil de Ejecutor de 
Prueba de Software.

Perfil de Ejecutor de Pruebas.
Debe conocer del ciclo de vida de una prueba, seguir un plan de ejecución y ciclos de prueba,
tener conocimiento en herramientas de gestión tipo ALM, interpretar diseño de casos de prueba,
solicitar datos para la prueba, determinar si el caso de prueba es exitoso, fallido, bloqueado 
o detenido por error, interpreta indicadores y reportes de prueba, escribe defectos y conoce 
el ciclo de vida de éstos, documenta evidencias de prueba, conoce manejadores de base de datos 
y sql simple. Sigue proceso de prueba estalecido por la compañía agile o tradicional.

Actividades de un ejecutor de pruebas.
Validar los accesos a los casos de prueba que previamente realizó un ingeniero de prueba.
Revisa tener los datos para la prueba.
Revisar plan de ejecución.
Esperar la señal del líder de prueba para empezar a probar.
Ejecutar los casos de prueba siguiendo los scripts.
Documentar los resultados en herramienta o matriz de casos de prueba.
Grabar la evidencia y almacenar en ruta oficial del proyecto o herramienta.
Si hay un defecto, lo registra y envía al área de desarrollo, de lo contrario pasa el caso de
prueba y si hay dudas en la ejecución, hay que revisarlas con el líder de prueba.

Un ejecutor de pruebas de software cuenta con insumos y entregables.
INSUMOS
Repositorio de ejecución.
Tipo de ciclo.
Ruta de evidencias de ejecución.
Plan de prueba.
Proceso y procedimientos.
Reglas del proyecto y ejecución.
Ruta casos de prueba en herramienta o matrices de prueba.
Rutas del aplicativo a ejecutar con usuarios y claves de acceso.

ENTREGABLES
Evidencias de ejecución.
Casos de prueba actualizados en herramienta o matrices de prueba.
Defectos documentados.

Prueba de software: es la validación y verificación de los requerimientos funcionales
Validación: comparamos el caso de prueba con el requerimiento.
verificación: comparamos el caso de prueba con el caso de uso (insumo inmediato).

Atributos de calidad de un sistema: amigable, no amigable, es parte de las pruebas de software.
El testing se hace con la intención de encontrar errores mediante el uso de procesos,
procedimientos, actividades definidas, estrategia y herramientas que ayuden a automatizar 
el proceso de la prueba.

Tipos de pruebas de software.
Prueba funcional.
PRUEBA MANUAL: es la prueba funcional que hacemos para validar que el sistema haga lo que tiene 
que hacer en apego a los requerimientos funcionales del usuario. Se lleva a cabo ejecutando 
acciones con el teclado o mouse para tocar los objetos de la pantalla y revisar las reacciones
del sistema.
PRUEBA AUTOMATIZADA: prueba funcional que programamos con lenguajes de programación asociados 
a marcos de trabajo llamados frameworks que permite automatizar las acciones manuales que un 
usario realiza en su aplicación o sistema y reproducirla "n" vecesmediante el uso de un robot, 
es más rápida que la manual.

Prueba no funcional: se enfoca a los atributos de calidad del producto.
DESEMPEÑO: es la prueba con un enfoque a los atributos del sistema, buscando leer el uso de 
recursos que el aplicativo utiliza para operar, CPU, ancho de banda, usuarios concurrentes, 
tiempos de respuesta y validando que no se degrade el sistema mientras se reliza este tipo de
pruebas. Se trata de validar el uso de recursos de un aplicativo mientras se está ejecutando con un 
cierto volumen de transacciones o usuarios concurrentes para ver cómo se comporta ese producto.
SEGURIDAD: prueba que utilizamos para detectar las vulnerabilidades de los sistemas WEB, se 
corren una serie de algoritmos predefinidos de hackeo para identificar los huecos en los
sistemas.

Ciclo de vida de un proyecto de prueba.
Análisis: analizar la factibilidad, identificar la estrategia y estimación para la atención 
del proyecto, se identifica el qué voy a hacer y se analizan las horas que tomará.
Planeción: se elabora un plan de trabajo, plan de pruebas. Identificar el qué, cómo, cuándo,
dónde lo vamos a trabajar el proyecto. Estas dos primeras etapas las hace el líder de pruebas.
Diseño: (ingeniero de pruebas) Empezamos a construir los casos de prueba (testware).
Ambiente: que todo este listo para las pruebas, hardware, software, hacer las pruebas de humo.
Ejecución: prueba dinámica, ejecutar los diseños tomando el testware diseñado previamente y 
empezamos a documentar defectos y evidencias de prueba.
Pruebas de aceptacion de usuario (UAT) User Acceptance Test: se ejecutan casos básicos, el 
usuario es el que valida las cosas, revisa que el sistema haga lo que el solicitó.
Cierre: se cierran repositorios, entregamos el proyecto de prueba, generamos el certificado
de prueba que avala el trabajo realizado durante la ejecución y UAT firmado por todos.

Caso de prueba.
El sistema se puede comparar con una caja negra, que le inyectaremos información, es decir 
entradas y las precondiciones es el estado en el que tiene que estar el sistema para capturar
la entrada. Las salidas es el evento o los datos que salen del sistema o mensaje de resultado.
La post condición es el estado en el que queda el sistema después de ejecutar la entrada.

Script de pruebas o procedimiento de pruebas.
Es el paso a paso o conjunto de pasos, para reproducir un caso de prueba. Es el medio para 
validar un caso de prueba y debe tener todos los detalles para poder ejecutarse sin ayuda del 
tester que lo creó, comunmente lo ejecuta el ingeniero de pruebas.

Reglas de un procedimiento de purebas: no contener datos duros (hardcode)
debe tener al menos seis pasos básicos:
1.- cómo acceder a la aplicación
2.- cómo colocar las precondiciones
3.- cómo ejecutar el caso
4.- cómo validar el resultado esperado
5.- cómo validar las post-condiciones
6.- cómo salir del sistema

Datos de pruebas.
Son aquellos datos que se requieren para ejecutar una prueba, datos requeridos para el set de 
prueba. Cada test case requiere de entrada: tipos de dato estáticos (no se queman) es decir 
es reutilizable como el número de una tarjeta de crédito y dinámicos se queman, es decir ya no 
se pueden volver a utilizar como el saldo de la tarjeta de crédito.

Testware.
Un proceso de negocio está constituido por requerimientos, un requerimiento puede tener de uno 
a n escenarios, de los escenarios se derivan los casos de pruebas, un escenario puede tener 
de uno a n casos de prueba. Los procedimientos de prueba son el paso a paso para ejecutar los 
casos.

Conjunto de casos de prueba: conjunto de casos de prueba relacionados que tienen características
similares. El objetivo es integrarlos en ciclos de prueba para su ejecución.

Ejecución de prueba.
Es tomar cada caso de prueba previamente diseñado por un tester analista o ingeniero de prueba 
y seguir el guión del procedimiento de prueba (script manual) para reproducirlo con los datos 
requeridos en esa casuística,siempre se documenta una evidencia de la ejecución de la prueba
independientemente del resultado. Los casos de prueba se agrupan y se ejecutan en ciclos de 
prueba. Hay una línea de tiempo para realizar esta ejecución, así como una duración estimada
para cada caso de prueba.

Parte del proceso donde se detona: entre la fase de diseño de prueba y la UAT. En la fase de 
ejecución de prueba está en su máximo punto.

Test set o grupo de casos de prueba: Conjunto de casos de prueba relacionados que tienen 
características similares. El objetivo es integrarlos en ciclos de prueba para su ejecución.

Ciclo de pruebas: es un conjunto de casos de prueba agrupados en un SET de pruebas que están 
listos para llevarlos a la ejecución, ya tienen un tester ejecutor asignado. El responsable de 
armar el ciclo de prueba es el líder de pruebas.

Estatus de un caso de pruebas.
No ejecutado: el caso de prueba no se ha iniciado en su ejecución (blanco).
Pasdo: cuando el resultado esperado es igual al actual que arrojó el sistema durante la 
ejecución (verde).
Fallado: el resultado esperado NO es igual al actual que arrojó el sistema durante la ejecución 
(rojo).
Detenido por error o bloqueado: debido a un caso de prueba fallado este caso no se puede 
ejecutar y se ejecutará en el siguiente ciclo de prueba (amarillo).

El tester ejecutor es quien va cambiando el estatus del caso de prueba durante su ejecución.
Este estatus se cambia en la herramieta de gestión de prueba o matriz de prueba.

Tipos de proyectos de prueba.
Nuevos: empezamos a desarrollar un producto de software desde cero, desarrollos grandes (3 
meses- 1 año).
Mantenimientos: modificaciones y actualizaciones a desarrollos existentes (semanas).
Incidentes: problemas productivos que se tienen que atender de forma inmediata, el mismo día.

Estatus de proyectos.
Pre-estimación: se hace la factibilidad del proyecto estimacion de la prueba.
Estimacion: ponemos en horas el esfuerzo.
Planificación: generamos plan de prueba, plan de trabajo.
Línea base: determinar que el estuerzo en horas, estimación se ha cerrado.
Abierto: cuando estamos trabajando en el proyecto.
Puede llegar a ser cancelado o suspendido.
Terminado: conclusión del proyecto (se han completado todas las tareas ) y está listo para irse
a producción.

Clasificación de proyectos.
Evolutivos: se requieren para operar la empresa.
Normativos: regulaciones de entidades externas gubernamentales.
Administrativos: proyectos internos para gestiones operacionales de la empresa.

Áreas de prueba.
Tenemos un coordinador de prueba que define las estrategias de cobertura anuales y establece 
las politicas de calidad en el area. Define el plan de capacitación anual y monitorea el 
desempeño de las sub-areas de prueba.

FUNCIONALES: valida los requerimientos funcionales del software, valida que el sistema haga lo 
que debe hacer y lo que no debe hacer.
NO FUNCIONALES: valida los requerimientos de calidad del software, que los tiempos de respuesta 
del sistema sean eficientes, mide la calidad, la seguridad, uso de recusos de un aplicativo 
por herramientas especializadas.
AMBIENTE: mantiene el ambiente de pruebas disponible para los proyectos, vigila la integridad 
del ambiente, que no entre cualquier desarrollador o tester para actualizar o crear datos en 
la base de datos. 

Proporciona datos para la prueba, da soporte antes, durante y al final de la prueba, genera 
accesos y permisos especiales para acceder al ambiente. Cuida correlaciones y administra las 
ventanas de tiempo de prueba, monitorea el desempeño de las áreas: funcional no funicional, 
ambiente, monitorea el uso de las herramientas, monitorea el apego a los procesos, vigila el 
uso de los recursos, lleva la facturación, vigila SLAs: niveles de servicio, que todos lleguen 
a tiempo en presupuesto. Ofrece indicadores para la toma de decisiones ya veces es mentor.
