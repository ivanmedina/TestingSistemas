2-Fundamentos del Ejecutor de Pruebas
	2.1-Qué es una Prueba de Softare
		2.1.1-Qué es Prueba de Software
			Es la Validación y verificación de los requerimientos funcionales atravez de una manera sistematica con el fin de encontrar errores usando procesos, procedimientosy herramientas para poder garantizar el exito.
		2.1.2-Tipos de Prueba
			Existen Pruebas:
			Funcional que se divide en:
			-Manual:
				Es lo que el sistema debe de hacer y lo que el sistema no debe de hacer.
			-Automatizada:
				Es la prueba que programamos con lenguajes asociados a los framewroks que replican los usos de un usuario realiza en su aplicacion n veces para poder ver el resultado acutal con el esperado.
			No Funcional que se divide en:
			-Desempeño:
				Es una prueba enfocada sobre los atributos del sistema para poder ver como se comportan los recursos para poder operar como el CPU, Ancho de Banda y funciona para ver que el sistema no falle durante la
				ejecucion para poder ver los limites del hardware en donde se utilizara el Software.
			-Seguridad:
				Son pruebas que se realiza para detectar las vulnerabilidades de los Sistemas Web y asi poder saber como atacarlas.
		2.1.3-Ciclo de Vida de una Prueba de Software
			Se divide en:
				-Analisis:
					Sirve para saber que voy a hacer y cuanto tiempo me tomara.
				-Planeación:
					En esto se identificara el que, cuando y donde y estas dos fases las Ejecuta el Lider de Pruebas.
				-Diseño:
					Aqui entra el Ing. de Prueba y se crean TestWare.
				-Ambiente:
					Todo lo relacionado a hardware y software, es la etapa final para poder hacer las pruebas y asi entrar a la fase de ejecucion sin problemas.
				-Ejecución:
					Se ejecutan los casos de prueba y se empiesa a documentar defectos y evidencias.
				-UAT(User Aceptan Test):
					Se ejecutan las pruebas con el usuario solo las mas importantes para que quede por terminado el proyecto.
				-Cierre:
					Se cierra todo la documentacion para poder entregar el certificado de que el software funciona chido y debe de estar firmado por todsos.
		2.1.4-Que es un Caso de Prueba:
			Se compone de Entradas/Precondiciones-Se ejecuta el programa-Salidas/Poscondiciones, en otras palabras se le asignan datos para ver si se ejecuta de manera correcta ya sea dando valores positivos o negativos
			para ver si el sistema hace lo que deberia de hacer.
		2.1.5-Que es un Script o Procedimiento
			Es la receta o formula para reproducir un caso de pruba y debe de ser legible para cualquier persona.
		2.1.6-Datos de Prueba
			Son aquellos que se necesitan para ejecutar una prueba, existen datos Estaticos(No se queman , no cambian), Dinamicos(Se quemas, cambian).
		2.1.7-Que es TestWare
			Permite reducir el costo de la realizacion.
	2.2-Que es Ejecucion de prueba
		2.2.1-Que esla ejecucion de la prueba
			Es tomar cada caso y seguir la receta para poder ver si funciona como se espera y si no funciona levantar un reporte, siempre llevar la evidencia.
		2.2.2-Parte del proceso donde se detona la prueba
			Se detona entre la fase de Diseño de Prueba y la UAT	
		2.2.3-Test Set o Grupo de Casos de Prueba
			Cojunto de casos de prueba similires, el objetivo es integrarlo en ciclos de prueba para la ejecucion.
		2.2.4-Que es un Ciclo de Pruebas
			Conjunto de SET  de Pruebas y la idea es ejecutar el 100% de los casos de prueba aunque no funcionen todos la calidad es diferente al progreso y lo realiza el ejecutor de pruebas.
		2.2.5-Estatus de un Caso de Prueba
			-No-Ejecutado:
				No se a echo nada(Blanco).
			-Pasado:
				El resultado es igual al esperado y se pasa al caso de prueba(Verde).
			-Fallado:	
				El resultado no es el esperado y se levanta un reporte o bug(Rojo).
			-Detenido por Error o Bloqueado:		
				No se puede ejecutar, se ejecuta en la siguiente ejecucion(Amarillo).
	2.3-Proyectos de Prueba
		2.3.1-Tipos de Proyectos de Prueba
			-Nuevos:
				Desarrollos grandes se inicia de cero y varia segun lo que se pida el tiempo de realizacion.
			-Mantenimietno:
				Se les da mantenimiento o actualizaciones que pueden durar semanas o menos.
			-Incidentes:	
				Problemas productivos se atienden al momento.
		2.3.2-Estatus de Proyectos
			-Pre-Estimacion:
				Estimacion de si funcionara el proyecto y tiempo.
			-Estimacion:
				Se marcan las horas problables que se utilizaran.
			-Planificacion:
				Se genera plan de prueba y trabajo.
			-Linea Base:
				Determina si lo sufrido esta terminado.
			-Abierto:
				El proyecto esta en uso.
			-Terminado:
				Fin del proyecto.
			-Suspendido:
				En lo que se consiguen recursos.
			-Cancelado:
				No hay recursos.

		2.3.3-Clasificacion de Pruebas
			-Evolutivos:
				Para operar la empresa.
			-Normativos:
				Regulacion por parte del gobierno y tener control.
			-Administrativos:
				Proyectos internos.
			
	2.4-Áreas de Prueba
		Area dirigida por el Coordinador de Pruebas que engloba las siguientes:
		2.4.1-Funcionales:
			Valida lo que el sistema debe de hacer y que no debe de hacer, valida los requerimientos pedidos.
		2.4.2-No Funcionales:
			Valida los requerimientos, mide la calidad, valida todos los atributos del sitema y que el uso de recursos sea eficiente y no la quiebre.
		2.4.3-Ambiente:
			Vigila la integridad, mantiene las pruebas disponibles.
		2.4.4-Control:
			Monitorea el desempeño de estas areas en tiempo y forma.
